# Evaluator 스킬 (생성자-평가자 패턴)

## 핵심 원칙
**작성 에이전트 != 검증 에이전트**
같은 에이전트가 작성하고 스스로 통과 판정하는 것을 금지한다.

## 트리거 조건
- 각 Step 파일 내부에 `# EVAL` 또는 `평가자 실행` 지시가 명시된 경우
- Step 번호 범위에 의존하지 않는다 — Step 파일이 직접 호출하는 방식이 기준

> **Why:** Step 번호 하드코딩 시 Step 파일이 추가/재번호되면 트리거 누락 발생

## 평가 프로토콜

### 1단계: 구현물 스크린샷 촬영

Playwright로 현재 구현 상태를 3가지 뷰포트(desktop 1920x1080, tablet 768x1024, mobile 390x844)로 캡처한다.

저장 경로: step_archive/screenshots/eval-rN-{viewport}.png

스크린샷은 반드시 `fullPage: true` 옵션으로 전체 페이지를 캡처한다.

**Playwright 실패 시 폴백:**
스크린샷 촬영이 실패하더라도 평가를 중단하지 않는다.
- 디자인 충실도 항목은 0점으로 기록하고 사유를 명시
- 나머지 3개 항목(기능 완성도, 코드 품질, 성능)은 코드 분석 기반으로 평가 진행
- 평가 결과에 `⚠️ 스크린샷 없음 - 코드 분석 기반 평가` 표기

### 2단계: 평가자 에이전트 스폰 (sonnet 모델)

평가자 에이전트에게 전달할 프롬프트:

> 너는 독립 평가자다. 코드를 수정하지 않는다. 판정만 한다.
> **절대 규칙: `.claude/` 디렉토리에 어떤 파일도 생성하지 않는다. 출력은 반드시 step_archive/outputs/ 에만 저장한다.**
>
> 1. 원본 참조 자료를 읽는다:
>    - step_archive/에서 해당 Step의 조사 결과 청크 파일
>    - step_archive/screenshots/research/ 의 원본 스크린샷
>
> 2. 구현 스크린샷을 읽는다:
>    - step_archive/screenshots/eval-rN-*.png
>
> 3. 이전 라운드 평가 결과를 읽는다 (있으면):
>    - step_archive/outputs/eval_rN-1.md
>    - 이전 FAIL 항목의 수정 여부 확인
>    - **점수 추이 비교**: 이전 라운드 총점과 현재 총점을 비교한다
>      - 점수가 하락하면 `⚠️ 점수 퇴보 감지 (이전 XX/40 → 현재 YY/40)` 경고를 결과에 명시
>      - 퇴보 원인 항목을 특정하여 피드백에 포함
>
> 4. 루브릭 기반 평가:
>    - 기능 완성도 (0-10): 요구사항 대비 구현 비율
>    - 디자인 충실도 (0-10): 원본 참조 대비 시각적 일치도
>    - 코드 품질 (0-10): 린팅/포매팅/타입 안전성
>    - 성능 (0-10): 로드 시간, DOM 복잡도
>
> 5. 결과 저장: step_archive/outputs/eval_rN.md
>    - 총점 40점 (만점): PASS
>    - 총점 40점 미만: FAIL + 구체적 피드백
>
> 형식:
>
> ## 평가 결과 Round N
> - 기능 완성도: X/10
> - 디자인 충실도: X/10
> - 코드 품질: X/10
> - 성능: X/10
> - 총점: XX/40
> - 판정: PASS/FAIL
>
> ### FAIL 항목 (있으면)
> 1. 항목명: 구체적 설명 + 수정 방향

### 3단계: 판정 처리
- **PASS**: 다음 Step으로 진행
- **FAIL**: 피드백을 구현 에이전트에 전달하여 수정 후 재평가 (최대 3라운드)

## 라운드 제한
- 최대 3라운드
- 3라운드 후에도 FAIL이면 실패 항목을 progress.json의 failed_steps에 기록하고 다음 Step으로 진행

# Step 61 - 포매팅 후 코드 무결성 검증 (독립 검증 루프)

포매팅 적용 후 코드가 정상적으로 동작하는지 검증한다.

## 핵심 원칙: 포매팅 에이전트 ≠ 검증 에이전트

Step 60에서 포매팅을 수행한 에이전트가 스스로 '문제없다'고 판단하는 것을 금지한다. 독립된 검증 에이전트가 무결성을 평가한다.

## 작업 흐름

### 에이전트 B (무결성 검증만, 코드 수정 금지)

1. **코드 무결성 검증**
   - 컴파일 언어: 빌드/컴파일 실행
   - 인터프리터 언어: 구문 검사 도구 실행

2. **기능 테스트**
   - 기존 테스트 스위트 실행
   - 실패 시 원인 분석 및 기록

3. **검증 결과 판정**
   - **PASS**: 모든 검증 통과 → 다음 단계로 진행
   - **FAIL**: 검증 실패 항목 기록 → 에이전트 A로 이동

### 에이전트 A (코드 수정만, 통과 판정 금지)

1. 에이전트 B의 FAIL 보고서 검토
2. 원인 분석 및 수정 (포매팅 버그 또는 기타 코드 문제)
3. 수정 완료 후 다시 에이전트 B로 전달 (검증 역할 자신이 수행 금지)

합리적인 선에서 최대한 많은 서브에이전트를 병렬로 사용하여 (동시 실행 최대 10개) 검증을 수행한다.

**포매팅 검증 단계에서 절대로 plan mode를 사용하지 않는다.**

**객관적 검증 기준:**
- 빌드/컴파일 성공 (컴파일 언어)
- 구문 검사 통과 (인터프리터 언어)
- 기존 테스트 통과 (테스트 존재 시)
- **c8 커버리지 유지**: `npx c8 --check-coverage --lines 80 --branches 70 --functions 80 vitest run` 으로 포매팅 후 커버리지가 하락하지 않았는지 확인
- **jscpd 중복률 유지**: `npx jscpd src/ --threshold 5 --reporters console` 으로 포매팅 후 중복률이 임계치를 초과하지 않았는지 확인. 초과 시 검증 실패
- **Semgrep 경고 유지**: `semgrep --config auto --json src/` 으로 포매팅 후 새로운 보안/품질 경고가 발생하지 않았는지 확인. 새 경고 발생 시 검증 실패

### 반복 제한: 무한 반복 + 스마트 탈출

회차 제한 없이 PASS가 나올 때까지 반복한다.

**탈출 조건:**
1. **PASS** → 즉시 종료
2. **동일 항목 3연속 [미수정]** → 해당 항목만 스킵 처리하고 나머지 계속
3. **모든 FAIL 항목이 스킵 상태** → 종료 (스킵 사유 기록)

## 실패 패턴 기록

종료 시 (PASS 또는 스킵) `step_archive/progress.json`의 `failure_patterns` 배열에 FAIL 항목을 추가한다.
- PASS로 종료된 경우에도 도중 FAIL이 있었던 항목은 기록한다.
- 형식: `{ "step": 61, "항목": "FAIL 항목 요약", "해결": true/false }`

**검증 결과는 청크 단위로 저장한다:**

```text
step061_포매팅검증_chunk1.md (500줄 이하)
step061_포매팅검증_chunk2.md (500줄 이하)
step061_포매팅검증_chunk3.md (500줄 이하)
...
```text

**작성 규칙**:
- 각 청크는 500줄 이하로 작성 (성능 최적화)
- `.claude/hooks/research-validator.ps1`에서 각 청크 검증 (BOM/CRLF/줄수/파일크기)
- 청크 그대로 유지 (병합 안 함)

에이전트 B(검증)와 에이전트 A(수정) 모두 haiku를 사용한다. (스크린샷 분석 없으므로 haiku)

# Step 102 - 접근성 테스트

## Step-Back

실행 전에 먼저 답하라:
- 이 테스트의 핵심 목적은? (한 문장)
- 테스트 실패 시 어느 Step으로 돌아가야 하는가?
- 반드시 확인해야 할 엣지 케이스 2가지는?

프로젝트 유형, 타겟 사용자층, 법적/산업 요구사항, UI 특성을 분석하여 접근성 기준·검사 범위·도구를 동적으로 결정한다.

결정된 기준에 따라 자동화 접근성 테스트를 구현하고, 위반 0건이 될 때까지 수정을 반복한다.

**이 단계에서 절대로 plan mode를 사용하지 않는다.**

**위반 0건 확인 없이 통과 처리 금지.**

## 사전 분석 (동적 결정)

1. **프로젝트 유형 파악**
   - 웹 앱 / 데스크톱 앱 / 모바일 앱 / CLI 등
   - 프로젝트 유형에 적합한 접근성 검사 도구 선택
   - 웹 앱이면 `@axe-core/playwright` 사용 (Step 4에서 이미 설치됨)
   - 웹 앱이 아니면 해당 플랫폼의 접근성 검사 도구 조사 후 선택

2. **타겟 사용자·법적 요구사항 분석**
   - 프로젝트 대상 지역·산업의 접근성 법규 파악 (ADA, EN 301 549, 장애인차별금지법 등)
   - 적용해야 할 WCAG 수준 결정 (A / AA / AAA)
   - 산업 특수 요구사항 확인 (공공기관, 금융, 교육 등)

3. **UI 특성 기반 검사 범위 결정**
   - step030_레이아웃설계_chunk*.md (레이아웃 설계)에서 UI 구조/컴포넌트 목록 추출
   - step030_전체설계_chunk*.md (전체 설계)에서 UI 구조/컴포넌트 목록 추출
   - 프로젝트에 실제 존재하는 UI 요소 기반으로 검사 항목 구성
   - 해당 없는 항목은 제외 (예: 영상 없으면 자막 검증 제외, 폼 없으면 폼 라벨 제외)

4. **검사 도구 태그·규칙 결정**
   - axe-core 사용 시: 분석 결과에 맞는 withTags 조합 결정
   - 프로젝트에 불필요한 규칙은 disableRules로 제외
   - 필요 시 커스텀 규칙 추가

## 검사 관점 (해당하는 항목만 동적 선택)

axe-core 등 자동화 도구가 검사할 수 있는 영역:

- **색상 대비** — 텍스트/배경 대비 비율 기준 미달
- **대체 텍스트** — img alt 누락, 장식 이미지 구분
- **ARIA 속성** — role/aria-label/aria-describedby 정합성
- **폼 라벨** — input-label 연결
- **헤딩 순서** — 계층 구조 위반
- **키보드 접근** — 인터랙티브 요소 키보드 도달 불가
- **빈 요소** — 텍스트 없는 button, a 태그
- **랜드마크** — 시맨틱 태그 부재
- **언어 속성** — html lang 누락/불일치
- **링크 목적** — 동일 텍스트 다른 대상

## 검증 절차

1. 사전 분석 결과를 기반으로 검사 스크립트 동적 작성
2. 모든 페이지/뷰/라우트에 대해 검사 실행 → 위반 목록 출력
3. **위반이 있으면:**
   - 각 위반의 impact(critical/serious/moderate/minor) 확인
   - 해당 요소의 소스 코드 수정
   - 재검사 실행
   - **위반 0건이 될 때까지 무한 반복**

### 반복 제한: 무한 반복 + 스마트 탈출

회차 제한 없이 PASS가 나올 때까지 반복한다.

**탈출 조건:**
1. **PASS** -> 즉시 종료
2. **동일 항목 3연속 [미수정]** -> 해당 항목만 스킵 처리하고 나머지 항목은 계속 반복
3. **모든 FAIL 항목이 스킵 상태** -> 종료 (해결 불가 판정, 스킵 사유를 최종 보고에 기록)

## 실패 패턴 기록

종료 시 (PASS 또는 스킵) `step_archive/progress.json`의 `failure_patterns` 배열에 FAIL 항목을 추가한다.
- PASS로 종료된 경우에도 도중 FAIL이 있었던 항목은 기록한다.
- 형식: `{ "step": 102, "항목": "FAIL 항목 요약", "해결": true/false }`

4. 위반 0건 확인 후 추가 검증 진행

## 🚨 WCAG 대비율 전수 검증 (axe-core 이후 필수 실행)

axe-core는 CSS 변수, 동적 인라인 스타일, striped 테이블 등에서 대비율 위반을 놓칠 수 있다.
Playwright `getComputedStyle`로 **브라우저가 실제 렌더링한 색상**을 기반으로 전수 검증한다.

### 왜 필요한가

"텍스트가 안 보인다" 문제는 두 가지 원인이 있다:
- **밝은 배경 + 밝은 텍스트** (예: Mantine striped 기본값 + 흰색 텍스트)
- **어두운 배경 + 어두운 텍스트** (예: 다크 테마 배경 + 기본 검정 텍스트)

눈으로 구분하지 말고 **숫자로 검증**한다.

### 참조 청크

모든 Playwright 스크립트는 `step_archive/step103.md`(참조 Step)에 정리되어 있다. 실행 전에 해당 파일을 Read하여 코드를 그대로 복사·실행한다.

### 실행 순서

각 페이지/라우트마다 다음 순서로 수행한다.

| 단계 | 내용 | 참조 |
|---|---|---|
| 1 | 기본 대비율 계산 (모든 텍스트 요소) | §1 |
| 2 | 인터랙션 상태 검증 (hover / focus) | §2 |
| 3 | 배경 이미지·필터·블렌드 플래그 | §3 |
| 4 | pseudo-element 배경 검증 | §4 |
| 5 | 스크롤 영역 검증 (viewport 바깥) | §5 |
| 6 | 반응형 브레이크포인트 검증 (mobile/tablet/desktop) | §6 |
| 7 | 테마 전환 검증 (light/dark + `prefers-color-scheme`) | §7 |
| 8 | 멀티 브라우저 검증 (Chromium/Firefox/WebKit) | §8 |
| 9 | 비동기 로딩 대기 후 재검증 | §9 |
| 10 | 자동 검증 불가 항목 → 수동 검토 목록 출력 | §10 |

### 검증 대상

모든 라우트 + 모달/토글 상태:
- 각 페이지 기본 상태
- 모달 열린 상태 (편집, 추가 등 버튼 클릭 후)
- 토글 활성화 상태 (IP 제한 등)

### 판정 기준

| 텍스트 유형 | WCAG AA 최소 대비율 |
|---|---|
| 일반 텍스트 (24px 미만, 또는 18.67px 미만 + bold) | **4.5:1** |
| 대형 텍스트 (≥ 24px, 또는 ≥ 18.67px + font-weight ≥ 700) | **3.0:1** |
| **주의**: font-weight 600(semi-bold)은 large text 기준 미해당 → 4.5:1 적용 | |

### 애니메이션 / transition 중간 상태

CSS transition 도중의 색상은 시작/종료 상태만 검증한다:
- hover 전 상태 검증 → hover 후 transition 완료 대기(transition-duration + 50ms) → hover 상태 검증
- 중간 프레임은 자동 검증 불가 → transition에 색상 변경이 포함된 요소 목록만 출력

### 실패 시

1. 실패 요소의 `fg`(텍스트색)와 `bg`(배경색) 확인
2. **둘 다 밝으면** → 배경을 어둡게 또는 텍스트를 어둡게
3. **둘 다 어두우면** → 텍스트를 밝게 또는 배경을 밝게
4. 수정 후 재검증 → **실패 0건까지 반복**

### ❌ 금지

- "눈으로 봤을 때 괜찮아 보인다"는 판단으로 통과 처리 금지
- 대비율 미달 요소를 무시하고 넘어가기 금지
- 밝은+밝은 / 어두운+어두운 원인 구분 없이 일괄 수정 금지
- 자동 검증 불가 목록(§10)을 확인하지 않고 넘어가기 금지

## 추가 검증 (자동화 도구가 못 잡는 항목)

자동 검사 통과 후, 프로젝트에 해당하는 항목만 Playwright/수동으로 추가 확인한다:

- 스크린 리더 탐색 순서가 논리적인지
- 포커스 트랩이 모달/다이얼로그에서 정상 작동하는지
- 동적 콘텐츠 변경 시 aria-live가 적절히 알림하는지
- 커스텀 위젯의 키보드 인터랙션 패턴이 WAI-ARIA Authoring Practices에 부합하는지
- 스킵 네비게이션이 작동하는지

해당 없는 항목은 건너뛴다.

## AI Slop 상태·인터랙션 검증 (CLAUDE.md 상속)

자동 접근성 검사와 별개로, 다음 상태/컴포넌트가 누락되지 않았는지 Playwright로 검증한다.

1. **Hover / Focus / Active 상태 존재 여부**
   - 모든 `a`, `button`, `[role="button"]`, 폼 요소에 `:hover`, `:focus-visible`, `:active` 시각 변화가 있는지 Playwright로 상태 전환 후 스크린샷 비교.
   - 포커스 링(`focus:ring-2` 상당) 누락 시 FAIL.
2. **트랜지션 일관성**
   - 상태 변화의 transition-duration이 프로젝트 전역 기준(예: 300ms)과 일치하는지.
   - 목적 없는 bounce/대형 모션 발견 시 FAIL.
3. **Empty State 컴포넌트**
   - 주요 리스트/테이블/카드 컨테이너에 데이터 0개 상태를 강제 렌더링(예: API를 빈 배열로 목킹)하여:
     - 안내 텍스트, 아이콘/일러스트, CTA 버튼이 중앙 배치되어 있는지.
     - 레이아웃이 붕괴하거나 빈 여백만 남지 않는지.
   - 하나라도 누락 시 FAIL.
4. **Tab index 순서**
   - 키보드 `Tab`으로 순회하며 논리적 순서(좌→우, 위→아래)를 유지하는지 확인.

합리적인 선에서 최대한 많은 서브에이전트를 병렬로 사용한다 (동시 실행 최대 10개).

**검증:**
- `.claude/hooks/accessibility-validator.ps1`에서 자동 검증

**검증 실패 시:**
- 위반 사항 분석
- 접근성 문제 수정
- 검증 통과할 때까지 반복

서브에이전트는 항상 haiku를 사용한다.

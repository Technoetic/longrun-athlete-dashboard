# Step 73 - 최적화 스크린샷 검증 (독립 검증 루프)

## 실행 내용

step072 최적화 구현 완료 후, 최적화로 인해 UI가 깨지지 않았는지 스크린샷으로 검증한다.

**필요한 파일:**

- step030_레이아웃설계_chunk*.md (레이아웃 설계)

**이 단계에서 절대로 plan mode를 사용하지 않는다.**

**스크린샷 없이 통과 처리 금지. 반드시 Claude가 직접 스크린샷을 눈으로 확인한다.**

## 핵심 원칙: 작성 에이전트 ≠ 검증 에이전트

- **에이전트 B (검증 전문)**: Sonnet - 스크린샷 검증, 문제 판정, 합격/불합격 결정만 담당. **코드 수정 금지**
- **에이전트 A (수정 전문)**: Sonnet - 에이전트 B의 판정에 따라 코드만 수정. **합격/불합격 판정 금지**

이 분리를 통해 검증자의 편견 없는 판정과 수정자의 빠른 피드백 루프를 보장한다.

---

## 검증 절차

### 1단계: 설계 명세 추출

step030_레이아웃설계_chunk*.md에서 레이아웃 구조, 배치, 비율, 반응형 브레이크포인트 등 설계 명세를 추출한다.

### 2단계: 스크린샷 촬영

개발 서버를 실행하고 Playwright로 주요 뷰포트별 스크린샷을 촬영한다.

### 3단계: 에이전트 B의 역할 (검증)

스크린샷을 직접 Read하여 설계 명세와 대조하고 **PASS/FAIL 판정만 수행한다**:
- 최적화로 인해 깨진 부분 식별
- 모든 뷰포트에서 레이아웃이 설계와 일치하는지 확인
- 판정 결과:
  - **PASS**: 설계와 일치 → 해당 항목 종료
  - **FAIL**: 깨진 부분을 구체적으로 기록하여 에이전트 A에 전달

### 4단계: 에이전트 A의 역할 (수정)

1. 에이전트 B의 FAIL 판정과 구체적 지적사항을 받는다.
2. **코드만 수정한다** (HTML/CSS/JS):
   - 에이전트 B가 지적한 부분에 집중
3. 스크린샷 재촬영
4. 재촬영한 스크린샷을 에이전트 B에 전달

### 5단계: 반복 제한: 무한 반복 + 스마트 탈출

회차 제한 없이 PASS가 나올 때까지 반복한다.

**탈출 조건:**
1. **PASS** → 즉시 종료
2. **동일 항목 3연속 [미수정]** → 해당 항목만 스킵 처리하고 나머지 계속
3. **모든 FAIL 항목이 스킵 상태** → 종료 (스킵 사유 기록)

## 실패 패턴 기록

종료 시 (PASS 또는 스킵) `step_archive/progress.json`의 `failure_patterns` 배열에 FAIL 항목을 추가한다.
- PASS로 종료된 경우에도 도중 FAIL이 있었던 항목은 기록한다.
- 형식: `{ "step": 73, "항목": "FAIL 항목 요약", "해결": true/false }`

## 검증 축 생성 규칙

**사전에 고정된 검증 항목 목록을 사용하지 않는다.**

1. step030_레이아웃설계_chunk*.md의 설계 명세에서 검증해야 할 시각적 요소를 식별한다
2. 식별된 요소마다 하나의 검증 축을 생성한다
3. 설계에 명시되지 않은 항목을 임의로 추가하지 않는다

## 디자인 자연스러움 검토

설계 명세 일치 여부와 별개로, 스크린샷을 보고 **디자인적으로 어색한 부분**이 없는지 추가 검토한다.

**최대한 섬세하게 파악한다. 사소한 위화감도 놓치지 않는다.**

**검토 축 생성 규칙:**

- 사전에 고정된 검토 항목 목록을 사용하지 않는다
- 스크린샷에서 실제로 보이는 시각 요소를 하나씩 식별한다
- 식별된 요소마다 "이 요소가 시각적으로 자연스러운가?"를 판단하는 검토 축을 생성한다
- 스크린샷에 없는 요소에 대한 검토 축을 임의로 추가하지 않는다

**문제 발견 시 (에이전트 B 판정 → 에이전트 A 수정):**

1. 에이전트 B가 어색한 부분을 구체적으로 기록 (어떤 요소가 왜 어색한지)
2. 에이전트 A가 해당 소스 코드(HTML/CSS/JS) 수정
3. 에이전트 A가 스크린샷 재촬영
4. 에이전트 B가 다시 직접 확인
5. **자연스럽고 어색하지 않은 디자인이 될 때까지 반복** (스마트 탈출 조건 동일)

합리적인 선에서 최대한 많은 서브에이전트를 병렬로 사용한다 (동시 실행 최대 10개).

- **에이전트 B (검증)**: sonnet 사용 - 스크린샷 검증 담당
- **에이전트 A (수정)**: sonnet 사용 - 코드 수정 담당

# Step 113 - 수정 후 디자인 시각 검증 (독립 검증 루프)

Step 112 구현 완료 후, 수정된 부분과 그 주변 UI가 시각적으로 어색하지 않은지 검증한다.

Step 89와 동일한 검증 태도와 절차를 따르되, 수정 영역에 집중한다.

**이 단계에서 절대로 plan mode를 사용하지 않는다.**

**스크린샷 없이 통과 처리 금지. 반드시 Claude가 직접 스크린샷을 눈으로 확인한다.**

## 핵심 원칙: 작성 에이전트 ≠ 검증 에이전트

- **에이전트 B (검증 전문)**: Sonnet - 스크린샷 검증, 문제 판정, 합격/불합격 결정만 담당. **코드 수정 금지**
- **에이전트 A (수정 전문)**: Sonnet - 에이전트 B의 판정에 따라 코드만 수정. **합격/불합격 판정 금지**

이 분리를 통해 검증자의 편견 없는 판정과 수정자의 빠른 피드백 루프를 보장한다.

---

## 🚨 검증 태도 (절대 준수)

Step 89의 검증 태도를 그대로 적용한다:

- **1px 단위까지 의심한다**
- **"거의 맞다"는 "틀렸다"와 같다**
- **빈 공간을 의심한다**
- **일반 사용자의 첫인상으로 본다**
- **스크린샷을 확대해서 본다**
- **비교 검증한다**
- **어색함의 원인을 정확히 특정한다**

## 검증 범위

1. **수정된 컴포넌트/페이지** — 변경된 UI 요소 직접 확인
2. **수정 영향 범위** — 수정된 요소 주변의 레이아웃, 간격, 정렬이 깨지지 않았는지 확인
3. **전체 페이지 맥락** — 수정된 부분이 전체 페이지와 시각적으로 조화되는지 확인

## 검증 절차

### 1단계: 스크린샷 촬영

Playwright로 수정 관련 페이지/뷰의 스크린샷 촬영

### 2단계: 에이전트 B의 역할 (검증)

스크린샷을 직접 Read하여 시각적으로 확인하고 **PASS/FAIL 판정만 수행한다**:
- 검증 태도(Step 89 준용)를 따라 최대한 예민하게 검증
- 수정된 컴포넌트/페이지 확인
- 수정 영향 범위 확인
- 전체 페이지 맥락 확인
- 판정 결과:
  - **PASS**: 모든 항목 통과 → 종료
  - **FAIL**: 어떤 요소가 어떻게 어색한지 구체적으로 기록하여 에이전트 A에 전달

### 3단계: 에이전트 A의 역할 (수정)

1. 에이전트 B의 FAIL 판정과 구체적 지적사항을 받는다.
2. **코드만 수정한다** (HTML/CSS/JS):
   - 에이전트 B가 지적한 부분에 집중
3. 스크립트 재실행 → 스크린샷 재촬영
4. 재촬영한 스크린샷을 에이전트 B에 전달

### 4단계: 반복 제한: 무한 반복 + 스마트 탈출

회차 제한 없이 PASS가 나올 때까지 반복한다.

**탈출 조건:**
1. **PASS** → 즉시 종료
2. **동일 항목 3연속 [미수정]** → 해당 항목만 스킵 처리하고 나머지 계속
3. **모든 FAIL 항목이 스킵 상태** → 종료 (스킵 사유 기록)

## 실패 패턴 기록

종료 시 (PASS 또는 스킵) `step_archive/progress.json`의 `failure_patterns` 배열에 FAIL 항목을 추가한다.
- PASS로 종료된 경우에도 도중 FAIL이 있었던 항목은 기록한다.
- 형식: `{ "step": 112, "항목": "FAIL 항목 요약", "해결": true/false }`

### 최종 확인

모든 항목 통과 확인 후 다음 단계 진행

합리적인 선에서 최대한 많은 서브에이전트를 병렬로 사용한다 (동시 실행 최대 10개).

- **에이전트 B (검증)**: sonnet 사용 - 스크린샷 검증 담당
- **에이전트 A (수정)**: sonnet 사용 - 코드 수정 담당

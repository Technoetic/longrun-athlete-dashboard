# Step 98 - 유닛 테스트 커버리지 검증

## Step-Back

실행 전에 먼저 답하라:
- 이 테스트의 핵심 목적은? (한 문장)
- 테스트 실패 시 어느 Step으로 돌아가야 하는가?
- 반드시 확인해야 할 엣지 케이스 2가지는?

프로젝트의 비즈니스 로직, 유틸리티 함수, 데이터 처리 로직에 대한 유닛 테스트를 작성하고 커버리지 목표를 달성한다.

프로젝트의 기술 스택(package.json)을 분석하여 Jest 또는 Vitest 중 적합한 테스트 러너를 동적으로 선택한다.

**이 단계에서 절대로 plan mode를 사용하지 않는다.**

**커버리지 목표 미달 시 통과 처리 금지.**

## 사전 분석 (동적 결정)

1. **프로젝트 기술 스택 파악**
   - `package.json`에서 프레임워크 확인 (React, Vue, Svelte, vanilla 등)
   - 이미 설치된 테스트 도구가 있는지 확인 (jest, vitest, mocha 등)
   - Vite 기반이면 Vitest, Webpack/CRA 기반이면 Jest 선택
   - 둘 다 아니면 프로젝트 특성에 맞는 도구 선택

2. **테스트 대상 파일 식별**
   - `src/` 내 순수 함수, 유틸리티 모듈, 데이터 변환 로직 우선
   - UI 컴포넌트보다 **비즈니스 로직 함수**를 우선 테스트
   - 테스트 불필요 파일 제외: 타입 정의(.d.ts), 상수 파일, 단순 re-export

3. **커버리지 목표 결정**
   - 프로젝트 규모와 로직 복잡도에 따라 목표 동적 설정
   - 최소 기준: Statements 70%, Branches 60%, Functions 70%, Lines 70%
   - 로직 집약 프로젝트는 더 높은 기준 적용 가능

## 실행 절차

### 1단계: 테스트 환경 구성

1. 선택한 테스트 러너 설치 (이미 있으면 건너뜀)
2. 커버리지 설정 추가 (c8 또는 istanbul)
3. `package.json`에 테스트 스크립트 추가
4. 필요 시 테스트 설정 파일 생성 (jest.config / vitest.config)

### 2단계: 유닛 테스트 작성

테스트 대상 함수별로:

1. **정상 케이스** — 기대 입력에 대한 올바른 출력
2. **경계 케이스** — 빈 배열, 0, null, 빈 문자열, 최대/최소값
3. **에러 케이스** — 잘못된 입력에 대한 적절한 에러 처리
4. **타입 변환** — 문자열↔숫자, 날짜 파싱 등 변환 로직

테스트 파일 위치: 테스트 대상과 같은 디렉토리에 `*.test.ts` 또는 `*.spec.ts`

### 3단계: 테스트 실행 및 커버리지 측정

```bash
## Vitest 예시
npx vitest run --coverage

## Jest 예시
npx jest --coverage
```text

### 4단계: 커버리지 미달 시 반복

1. 커버리지 리포트에서 미커버 라인/분기 확인
2. 해당 함수에 대한 추가 테스트 케이스 작성
3. 재실행 → **목표 달성까지 반복**

## 테스트 품질 기준

- ❌ 구현 세부사항 테스트 금지 (내부 변수명, 호출 순서 등)
- ❌ 외부 API/DB 직접 호출 금지 (mock 사용)
- ❌ 스냅샷 테스트 남용 금지
- ✅ 함수의 입출력 계약(contract) 검증
- ✅ 각 테스트는 독립적이고 순서 무관하게 실행 가능
- ✅ 테스트 이름은 "무엇을 하면 어떤 결과가 나오는지" 명확히 기술

## ❌ 금지

- 커버리지 목표 미달 상태로 통과 처리 금지
- 테스트 없이 "로직이 단순하므로 불필요" 판단 금지
- `/* istanbul ignore */` 또는 `/* c8 ignore */`로 커버리지 회피 금지
- any 타입으로 테스트 타입 검사 우회 금지

합리적인 선에서 최대한 많은 서브에이전트를 병렬로 사용한다 (동시 실행 최대 10개).

서브에이전트는 항상 haiku를 사용한다.

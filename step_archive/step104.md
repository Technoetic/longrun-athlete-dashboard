# Step 104 - 번들 사이즈/성능 예산 검증

빌드 산출물의 크기를 측정하고 성능 예산 기준을 초과하지 않는지 검증한다.

프로젝트의 빌드 도구(Vite, Webpack, esbuild 등)를 분석하여 적합한 측정 방법을 동적으로 결정한다.

**이 단계에서 절대로 plan mode를 사용하지 않는다.**

**성능 예산 초과 시 통과 처리 금지.**

## 사전 분석 (동적 결정)

1. **빌드 도구 파악**
   - `package.json`에서 빌드 스크립트 및 번들러 확인
   - Vite → `rollup-plugin-visualizer` 또는 `vite-bundle-analyzer`
   - Webpack → `webpack-bundle-analyzer`
   - 기타 → `source-map-explorer` 또는 빌드 출력 직접 분석

2. **프로젝트 유형별 성능 예산 결정**
   - SPA / 정적 사이트 / 대시보드 등 프로젝트 성격 파악
   - 성능 예산 기준 동적 설정:
     - **총 번들 크기** (gzip 기준): 프로젝트 유형에 따라 적정 기준 결정
     - **개별 chunk 최대 크기**: 단일 chunk가 과도하게 큰지 확인
     - **초기 로드 JS**: 첫 화면에 필요한 JS 크기
     - **CSS 총 크기**: 스타일시트 전체 크기

3. **측정 대상 확인**
   - `dist/`, `build/`, `.next/` 등 빌드 출력 디렉토리 탐색
   - JS, CSS, 이미지, 폰트 등 에셋 유형별 분류

## 실행 절차

### 1단계: 프로덕션 빌드 실행

```bash
## package.json의 build 스크립트 실행
npm run build
```text

### 2단계: 번들 크기 측정

1. 빌드 출력 디렉토리의 모든 파일 크기 측정 (원본 + gzip)
2. 에셋 유형별 분류 및 합산
3. chunk별 크기 분석

### 3단계: 번들 구성 분석

1. 번들 분석 도구로 각 chunk에 포함된 모듈/라이브러리 시각화
2. 비정상적으로 큰 의존성 식별
3. 중복 포함된 라이브러리 감지
4. tree-shaking 실패 모듈 확인

### 4단계: 성능 예산 초과 시 최적화

초과 항목이 있으면:

1. **원인 분석** — 어떤 모듈/라이브러리가 크기를 키우는지 특정
2. **최적화 적용**:
   - 불필요한 의존성 제거
   - 동적 import로 코드 스플리팅
   - 대형 라이브러리를 경량 대안으로 교체
   - 이미지/폰트 최적화
3. **재빌드 → 재측정 → 기준 달성까지 반복**

## 검증 항목

- ✅ 총 JS 번들 크기 (gzip) 가 예산 이내
- ✅ 총 CSS 크기가 예산 이내
- ✅ 단일 chunk가 과도하게 크지 않음
- ✅ 중복 포함된 라이브러리 없음
- ✅ tree-shaking이 정상 작동 (사용하지 않는 export가 번들에 미포함)
- ✅ 소스맵이 프로덕션 번들에 포함되지 않음 (.map 파일 분리 또는 미생성)

## ❌ 금지

- 성능 예산 기준 초과 상태로 통과 처리 금지
- 측정 없이 "작아 보이므로 괜찮다" 판단 금지
- 소스맵을 비활성화하여 크기를 줄이는 꼼수 금지
- 기능 제거로 크기를 줄이는 것 금지 (최적화만 허용)

합리적인 선에서 최대한 많은 서브에이전트를 병렬로 사용한다 (동시 실행 최대 10개).

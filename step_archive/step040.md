# Step 40 - 조사 스크린샷 vs 구현 스크린샷 비교 검증 (독립 검증 루프)

## 실행 내용

Step 39 레이아웃 검증 통과 후, **조사 단계에서 촬영한 참고 스크린샷**과 **현재 구현 스크린샷**을 나란히 비교하여 디자인 품질을 검증한다.

**필요한 파일:**

- `step_archive/screenshots/research/awwwards-*.png` (Step 22 Awwwards 레이아웃 조사 스크린샷 — 비교 기준)
- step030_레이아웃설계_chunk*.md (레이아웃 설계)

**이 단계에서 절대로 plan mode를 사용하지 않는다.**

**스크린샷 없이 통과 처리 금지. 반드시 Claude가 직접 스크린샷을 눈으로 확인한다.**

## 핵심 원칙: 작성 에이전트 ≠ 검증 에이전트

- **에이전트 B (검증 전문)**: Sonnet - 스크린샷 비교, 문제 판정, 합격/불합격 결정만 담당. **코드 수정 금지**
- **에이전트 A (수정 전문)**: Sonnet - 에이전트 B의 판정에 따라 코드만 수정. **합격/불합격 판정 금지**

이 분리를 통해 검증자의 편견 없는 판정과 수정자의 빠른 피드백 루프를 보장한다.

## 검증 절차

### 1단계: Awwwards 기준 스크린샷 수집

1. `step_archive/screenshots/research/awwwards-*.png` 목록을 수집한다.
2. 각 스크린샷을 Read하여 어떤 Awwwards 수상작인지 파악한다.

### 2단계: 구현 스크린샷 촬영

1. 개발 서버를 실행한다.
2. Playwright로 주요 뷰포트별(데스크톱, 태블릿, 모바일) 스크린샷을 촬영한다.
3. `step_archive/screenshots/compare-impl-desktop.png`, `compare-impl-tablet.png`, `compare-impl-mobile.png` 로 저장한다.

### 3단계: 비교 검증 루프

#### 3-1. 에이전트 B의 역할 (검증)

**각 Awwwards 스크린샷에 대해 다음을 반복한다:**

1. Awwwards 스크린샷을 Read한다.
2. 구현 스크린샷을 Read한다.
3. 다음 축으로 비교하여 **PASS/FAIL 판정만 수행한다**:
   - **레이아웃 구조**: 전체 배치, 영역 분할, 비율이 Awwwards 수상작의 우수한 패턴을 반영했는가
   - **시각적 계층**: 정보 계층(제목, 본문, 부가정보)이 명확히 구분되는가
   - **여백/간격**: 요소 간 여백이 Awwwards 수준으로 충분하고 균일한가
   - **색상/대비**: 가독성과 심미성이 Awwwards 수준에 도달했는가
   - **컴포넌트 완성도**: 버튼, 카드, 패널 등 개별 요소가 Awwwards 수준으로 다듬어졌는가
   - **컨트롤 패널 배치**: Awwwards 사례에서 파악한 최적 위치에 배치되었는가
   - **시각화 영역 비율**: 메인 콘텐츠 영역이 충분한 공간을 확보했는가
   - **반응형 전환**: 뷰포트별 레이아웃 전환이 Awwwards 수상작처럼 자연스러운가

4. **판정 결과:**
   - **PASS**: 모든 축이 Awwwards 수준 달성 → 해당 항목 종료
   - **FAIL**: 부족한 부분을 구체적으로 기록하여 에이전트 A에 전달

#### 3-2. 에이전트 A의 역할 (수정)

1. 에이전트 B의 FAIL 판정과 구체적 지적사항을 받는다.
2. **코드만 수정한다** (HTML/CSS/JS):
   - 에이전트 B가 지적한 부분에 집중
   - Awwwards 수상작의 우수 패턴 반영
3. 스크린샷 재촬영
4. 재촬영한 스크린샷을 에이전트 B에 전달

#### 3-3. 반복 제한: 무한 반복 + 스마트 탈출

회차 제한 없이 PASS가 나올 때까지 반복한다.

**탈출 조건:**
1. **PASS** → 즉시 종료
2. **동일 항목 3연속 [미수정]** → 해당 항목만 스킵 처리하고 나머지 계속
3. **모든 FAIL 항목이 스킵 상태** → 종료 (스킵 사유 기록)

## 통과 조건

- 모든 비교 축에서 Awwwards 수상작 수준의 품질에 도달
- 레이아웃 구조가 Awwwards 사례의 우수 패턴을 반영
- 수정→재촬영→재비교 루프가 더 이상 문제를 발견하지 않음

합리적인 선에서 최대한 많은 서브에이전트를 병렬로 사용한다 (동시 실행 최대 10개).

- **에이전트 B (검증)**: sonnet 사용 - 스크린샷 비교·분석 담당
- **에이전트 A (수정)**: sonnet 사용 - 코드 수정 담당

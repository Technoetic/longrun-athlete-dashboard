## Phase 진도 (실제 18 커밋)

### Phase 0 — 프로토콜 해독 (e4f0b1f ~ 7be153b)

핵심 발견:
- **VeryFit 앱(`com.watch.life`)이 릴리즈에도 디버그 로그를 그대로 출력**. 이게 Ghidra 역분석을 live traffic 관찰로 대체시킨 최대 돌파구.
- GATT 프로파일: service `00000af0`, write `af6/af1`, notify `af7/af2`, CCCD `2902`.
- Legacy v1/v2 프레이밍: **raw body, no magic, no CRC, no length prefix**. BLE write length가 곧 패킷 길이.
- v3 magic 프레이밍: `33 DA AD DA AD 01 [len LE16] [cmd] [sub] [nseq LE16] [payload] [crc LE16]`. `len_field = total - 3`.
- **CRC16 = CCITT-FALSE (poly=0x1021, init=0xFFFF) over frame[1..total-2)**. 첫 magic 바이트 `0x33` skip하는 관습 때문에 표준 catalog 검색 실패했음. Differential brute force 로 찾음 (`scripts/crc16_bruteforce.py`).
- 205개 VBUS_EVT opcode, 121개 JNI 엔트리, 1330개 protocol 심볼 카탈로그.
- 법적 근거: **저작권법 §101조의4 상호운용성 예외**. Clean-room 프로세스 문서화.

산출물: `PHASE_0_REPORT.md`.

### Phase 1 M1-M4 — R21 직접 통신 (ce24151 ~ 3f06fec)

- **M1a (ce24151)**: `IdoBleClient.kt` 스켈레톤 (247 lines), UUIDs/cmds/GATT callback.
- **M1c (b22fb2f 포함)**: Live keepalive 왕복 성공. `02 01` 송신 → `02 01 6A 1F 01 01 02 64 ... 03` (20B) 응답 확인.
- **핵심 발견**: 바인딩 핸드셰이크 **불필요**. 최초 pair 후 어떤 앱이든 GATT 자유 접근 가능. (원래 M4로 예상했던 것이 M1에서 자동 해결)
- **M2 (b22fb2f 포함)**: 상시 HR 토글. `03 63 55 01 55 55 78 32 00 00 17 3B 08 07 00 00` 송신 → `07 40 02 00` ack + `03 63 00` success. 16B one-shot 명령.
- **M3α (b22fb2f 포함)**: Legacy `02 XX` sweep 8개 device-info 명령 매핑. `02 01` 응답의 offset 7이 **배터리 퍼센트**임을 발견 (0x64=100).
- **M3β (7be153b)**: v3 CRC16 해독 + `buildV3Frame` + `buildHealthQuery`. cat=0x01~0x10 전범위 응답. cat=0x08 daily summary에 오늘 날짜(EA 07 04 0F = 2026-04-15) + 4841 걸음 + 230 kcal 확인.
- **M4 (3f06fec)**: Multi-packet reassembly (0x33 continuation marker) + `IdoParser.parseDailySummary` + `HealthBridge.fetchDaily` → SyncWorker 통합 → **DB id=584 에 heart_rate=86, resting_heart_rate=86** 최초 저장. 대시보드 end-to-end 증명.

### Phase 2-A — IDO 값 우선 (69a6f29)

Health Connect가 주는 `steps=15594` 는 폰 + 여러 앱 합계라 부정확. R21 손목 실측 `steps=4841` 이 진짜. LongRun이 R21 생태계 대시보드이므로 IDO 값이 우선이어야 맞음. 정책 전환:

- `heart_rate`: IDO hr stream latest (가장 신선)
- `resting_heart_rate`: IDO daily rhr
- `steps`, `distance_km`, `exercise_minutes`, `active_calories`: IDO 덮어쓰기 (HC 값이 있어도)
- `basal_calories`: HC 유지 (R21에 기초대사 개념 없음)
- `sleep_hours`, `audio_db`: HC 유지

### Phase 2-C — 분단위 HR 스트림 파서 (eb06d84)

cat=0x03 917B payload에서 `[0x05, bpm]` 2바이트 pair를 defensive scan으로 추출. 364 샘플, min 79 / max 144 / avg 112. Segment marker (`1f`, `ff`, `06`, `51`, `4a`...) 는 slide-by-1로 회피.

중요한 의미론 분리:
- `heart_rate` = stream.latestBpm (지금 실시간)
- `resting_heart_rate` = daily summary rhr (baseline)

이전에는 둘이 같은 값이었는데 이제 **심박수 113 vs 안정심박 88** 로 의미 있게 분리.

## 미해결 / 연기 (Phase 2-B, 4-D)

**Phase 2-B — HRV/SpO2**:
- Phase 0 strings에서 `PROTOCOL_V3_HEALTH_DATA_TYPE_HRV`, `V3_support_set_spo2_all_day_on_off_03_44`, `spo2_add_notify_flag` 같은 심볼 확인.
- cat=0x01~0x20 sweep 에서 0x11-0x1D timeout → HRV/SpO2는 `cmd=0x04 sub=0x00` 경로로 fetch 안 됨.
- 추정 경로: **설정 명령 (`cmd=0x03 sub=0x44`) 로 상시 SpO2 on → notify push로 af2에서 수신**. 구현하려면:
  1. set 명령 바이트 역추적 (Phase 0 때 못 관찰한 부분)
  2. notify listener 추가 (지금은 pendingReply 하나만 지원)
  3. R21이 실제 측정한 이력이 있어야 값이 나옴 (며칠 대기)
- 예상 시간: 4~8시간 연구 + 며칠 대기. **세션 피로도에 비해 가치 낮아서 보류**.

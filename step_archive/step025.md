# Step 25 - 기획: 전체 조사결과 기반 (독립 검증 루프)

## Memory-of-Thought

기획 전에 다음을 확인하라:
- step_archive/progress.json의 failure_patterns — 이전에 실패한 패턴을 반복하지 않는다
- 이전 기획 Step의 결과 파일 — 중복되거나 상충되는 내용이 없는지 확인한다
- 성공한 기획 패턴이 있으면 재활용한다

## 실행 내용

step016 전체 조사결과를 기반으로 기획 문서 초안을 작성한다. **기획 작성자와 검증자를 분리하여** 조사 결과가 기획에 충실히 반영되었는지 확인한다.

## 핵심 원칙: 작성 에이전트 ≠ 검증 에이전트

기획을 작성한 에이전트가 스스로 "조사 결과를 잘 반영했다"고 판단하는 것을 **금지**한다.

**필요한 파일:**

- step016_조사결과_chunk*.md (전체 조사결과)

Class 지향으로 기획한다.

## 실행 순서

### 1단계: 기획 작성 에이전트 (에이전트 A) 실행

에이전트 A가 step016_조사결과_chunk*.md를 기반으로 기획 문서 초안을 작성한다.

결과: step025_planning_chunk*.md (500줄 이하/청크)

### 2단계: 조사 반영 검증 에이전트 (에이전트 B) 실행

**에이전트 B의 역할: 조사 결과가 기획에 반영되었는지 검증만 수행. 기획 수정 금지.**

에이전트 B에게 전달할 프롬프트 (N = 현재 라운드 번호):

```text
너는 조사 반영 검증자다. 기획을 수정하지 않는다. 반영 여부 판정만 한다.

1. 조사 결과를 모두 Read한다:
   - `step_archive/step016_조사결과_chunk*.md` (Glob 검색)
   - `step_archive/research-raw-*.txt` (Glob 검색하여 원본 데이터 확인)

2. 기획 문서를 모두 Read한다:
   - `step_archive/step025_planning_chunk*.md` (Glob 검색)

3. 이전 라운드 검증 결과가 있으면 Read한다:
   - `step_archive/outputs/step025_검증_r*.md` (Glob 검색)

4. 다음 축으로 검증한다:
   - **데이터 기반**: 기획의 각 결정이 조사 데이터에 근거하는가 (사전 지식 아닌 수집 데이터)
   - **누락**: 조사에서 발견된 중요 패턴이 기획에 빠지지 않았는가
   - **왜곡**: 조사 결과가 기획에서 잘못 해석/변형되지 않았는가
   - **출처**: 기획의 주요 결정마다 출처(조사 파일/스크린샷)가 추적 가능한가

5. 결과를 `step_archive/outputs/step025_검증_rN.md`에 저장한다.
   - 반영 충실하면: "PASS"로 시작
   - 부족하면: "FAIL"로 시작하고 누락/왜곡 항목 구체적 나열
```text

에이전트 B는 haiku를 사용한다.

### 3단계: 판정 확인

- **PASS** → 다음 Step으로 이동
- **FAIL** → 4단계 (기획 보강 에이전트 A)

### 4단계: 기획 보강 에이전트 (에이전트 A) 실행

에이전트 A가 검증 피드백을 반영하여 기획 문서를 보강한다. 통과 판정하지 않는다.

### 반복 제한: 무한 반복 + 스마트 탈출

**최대 5라운드**까지 반복한다. 5라운드 후에도 PASS가 아니면 남은 FAIL 항목을 모두 스킵 처리한다.

**탈출 조건:**
1. **PASS** → 즉시 종료
2. **동일 항목 2연속 [미수정]** → 해당 항목만 스킵 처리하고 나머지 항목은 계속 반복
3. **모든 FAIL 항목이 스킵 상태** → 종료 (해결 불가 판정, 스킵 사유를 최종 보고에 기록)

## 실패 패턴 기록

종료 시 (PASS 또는 스킵) `step_archive/progress.json`의 `failure_patterns` 배열에 FAIL 항목을 추가한다.
- PASS로 종료된 경우에도 도중 FAIL이 있었던 항목은 기록한다.
- 형식: `{ "step": 25, "항목": "FAIL 항목 요약", "해결": true/false }`

## 주의사항

- 에이전트 A가 스스로 "잘 반영했다"고 판단하는 것을 금지한다
- 에이전트 B가 기획을 수정하는 것을 금지한다
- 에이전트 B의 PASS/FAIL 판정만이 루프 종료 조건이다

합리적인 선에서 최대한 많은 서브에이전트를 병렬로 사용해야 한다 (동시 실행 최대 10개).

**기획 결과는 청크 단위로 저장한다:**

```text
step025_planning_chunk1.md (500줄 이하)
step025_planning_chunk2.md (500줄 이하)
step025_planning_chunk3.md (500줄 이하)
...
```text

서브에이전트는 항상 haiku를 사용한다.
